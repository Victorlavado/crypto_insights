# Q12 — Política de gap cuando un signal es None

## Contexto

El plan dice "fusión penaliza el composite score" cuando una fuente cae y el signal queda `None`, sin especificar cómo. Tres semánticas son mutuamente incompatibles:

- **(a) Renormalizar pesos sobre signals presentes** (None se ignora)
  - Pro: composite score sigue interpretable.
  - Contra: bias hacia signals supervivientes — un proyecto con solo 2 de 7 signals "vivos" puede dar score alto artificial.

- **(b) Tratar None como 0**
  - Pro: simple, explícito.
  - Contra: penaliza falsamente proyecto sano si fuente cae (Hyperliquid downtime → todos los proyectos parecen distribuyendo).

- **(c) Estado `degraded` + renormalización condicional**
  - Pro: honesto, da info accionable.
  - Contra: rompe la enumeración de 6 estados original; añade complejidad.

## Recomendación

**Híbrido**:
- Si <30% del peso total está faltante → renormalizar (a) y emitir warning visible en dashboard ("scores parciales: faltan funding, mindshare").
- Si ≥30% del peso total está faltante → estado `degraded`, `reason_code=GAP_DATOS`, `reason_data={missing_signals: ["funding", "mindshare"], total_weight_missing: 0.45}`. NO computar composite_score.
- Persistir flag boolean `has_gaps` en `PROJECT_STATE` para badge "datos parciales" siempre que haya algún gap, aunque <30%.

## Materialización

Como **ADR 0005 — Política de gap en signals** antes de implementar Fase 3.

## Acción esperada

Confirmar (c) híbrido, contrapropon, o validar que (a) puro es suficiente para MVP (con la aceptación de que dashboard puede mentir si una fuente cae).
